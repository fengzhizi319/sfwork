# 01 产品定位与用户场景

## 1.1 产品定位

**SecretPad 前端**是 SecretFlow 隐私计算生态的可视化控制台，面向企业级多方安全计算与可信执行环境（TEE）场景，提供从数据接入、项目协作、算法编排到模型发布的全链路 Web 化操作界面。

一句话定义：
> 让多方机构在浏览器里就能完成隐私计算项目的协作、编排、运行与治理。

## 1.2 目标用户

| 用户角色 | 主要职责 | 关注重点 |
|---|---|---|
| **平台管理员（Admin）** | 部署平台、注册节点、管理机构账号、监控系统健康 | 节点状态、路由连通、系统资源、审计日志 |
| **数据提供方（Data Owner）** | 注册数据源、上传数据、授权数据给项目、审批协作 | 数据安全、授权范围、审批消息 |
| **算法/建模工程师（Modeler）** | 创建项目、编排 DAG、运行训练、调参、查看结果 | 组件库、画布体验、任务状态、模型效果 |
| **项目发起方（Project Owner）** | 发起项目、邀请机构、管理训练流、发布模型 | 项目生命周期、参与方状态、版本管理 |
| **审计/合规人员（Auditor）** | 查看审批记录、结果产物、操作日志 | 消息中心、结果下载、周期任务 |

## 1.3 核心使用场景

### 场景 A：MPC 联邦建模（管道模式）
1. 管理员在中心平台注册多个边缘节点（如 alice、bob）。
2. 各机构在边缘节点上传本地数据表并授权到同一项目。
3. 建模工程师进入项目，从组件库拖拽数据读取、特征工程、安全统计、联邦学习组件，构建训练流。
4. 运行训练流，实时查看每个组件的运行状态与日志。
5. 训练完成后将模型提交到模型管理，发布为在线服务。

### 场景 B：TEE 枢纽建模
1. 机构选择 TEE 枢纽模式创建项目。
2. 将本地数据加密上传到 TEE 节点。
3. 在中心画布编排 TEE 可信计算组件。
4. 运行后从 TEE 节点下载结果，期间需通过 TEE 下载审批。

### 场景 C：P2P 自治协作
1. 无中心节点的机构间通过 AUTONOMY 模式直连。
2. 一方发起项目创建投票，其他参与方在消息中心同意/拒绝。
3. 项目审批通过后，各方才能开始 DAG 编排与训练。
4. 节点路由、数据授权同样需要通过投票审批。

## 1.4 设计原则

1. **降低隐私计算门槛**：把 Kuscia/SecretFlow 的 Domain/DomainData/Job 等概念映射为“节点/数据表/任务”等用户语言。
2. **多方协作可视化**：通过项目、消息中心、审批状态把跨机构的异步协作过程显性化。
3. **画布即 IDE**：DAG 画布需要像算法 IDE 一样支持组件拖拽、参数配置、运行调试、日志查看。
4. **权限最小化**：不同平台类型（CENTER/EDGE/AUTONOMY）看到不同的菜单和操作，避免误操作。
5. **状态可追踪**：任务、节点、数据、审批、模型都需要清晰的状态标签与进度反馈。

## 1.5 产品边界

当前前端不直接处理：
- 底层 Kuscia 集群的部署与运维（由后端/运维脚本负责）。
- SecretFlow 组件的内部实现（仅通过组件元数据渲染参数表单）。
- 模型在线服务的运行时监控详情（仅展示 Kuscia Serving 返回的服务状态与地址）。
