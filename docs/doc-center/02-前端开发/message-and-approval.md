# 页面需求：消息中心与审批

## 页面目标
集中展示项目邀约、数据授权、节点合作、TEE 下载、项目归档等需要用户处理的投票与通知，支持同意/拒绝操作。

## 用户故事
- 作为项目参与方，我希望在消息中心看到所有待我审批的事项。
- 作为项目发起方，我希望看到我发起的投票的状态。
- 作为数据提供方，我希望对数据授权请求进行审批。

## 主界面元素

### Tab 切换
- **我处理的**：需要当前用户/机构审批的消息。
- **我发起的**：当前用户/机构发起的投票。

### 筛选条件
- 消息类型：项目邀约、项目归档、数据授权、节点合作、TEE 下载等。
- 状态：全部 / 待处理 / 已处理。

### 消息列表
- 消息标题与类型图标。
- 发起方、接收方、发起时间。
- 状态标签：待处理 / 已同意 / 已拒绝。
- 操作：同意、拒绝。

### 消息详情抽屉
- 展示投票的完整信息、参与方投票进度。
- 拒绝时需填写拒绝理由。

## 操作与反馈

| 操作 | 反馈 |
|---|---|
| 同意 | 调用 `/message/reply` 提交同意，刷新列表与未处理数 |
| 拒绝 | 弹窗/抽屉填写理由后提交 |
| 查看详情 | 打开详情抽屉 |

## 业务规则
- 未处理消息数显示在顶部 Header 铃铛上，实时或定时刷新。
- 项目创建投票需所有受邀机构同意，项目才变为 `APPROVED`。
- 任一机构拒绝则项目进入 `ARCHIVED` 或 `REJECTED` 状态。
- 节点路由、数据授权、TEE 下载等投票有独立的 Handler 与状态机。

## 审批流程示意

```mermaid
sequenceDiagram
    actor Initiator as 发起方
    actor Participant as 参与方
    participant FE as 消息中心
    participant BE as SecretPad 后端
    participant Kuscia as Kuscia 路由/数据授权

    Initiator ->> BE: 发起审批（项目/路由/数据/TEE下载）
    BE ->> BE: 生成 VoteRequest + VoteInvite，签名
    BE -->> FE: 推送到参与方消息中心
    Participant ->> FE: 查看消息
    alt 同意
        Participant ->> BE: 同意 + 签名回复
        BE ->> Kuscia: 执行回调（创建路由/授权/允许下载）
        Kuscia -->> BE: 成功
        BE -->> FE: 更新状态
    else 拒绝
        Participant ->> BE: 拒绝 + 理由
        BE -->> FE: 标记拒绝，通知发起方
    end
```
