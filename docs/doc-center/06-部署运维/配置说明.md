## 2、为什么新增一个算法组件不需要改动 Kuscia 源码？

很多初次接触该项目的开发者会疑惑：新增一个 `privacy/l_diversity` 组件，为什么在 Kuscia 侧“什么都不用改”？这是因为 Kuscia 在整体架构中扮演的是**通用编排层**，而不是**算法语义层**。下面从协议、CRD、镜像、数据四个维度详细说明。

### 1. Kuscia 只传递通用任务配置，不解析组件语义

SecretPad 后端生成 `TaskInputConfig` 后，通过 KusciaAPI gRPC 提交给 Kuscia。`TaskInputConfig` 的核心字段是：

```json
{
  "sf_datasource_config": { ... },
  "sf_cluster_desc": { ... },
  "sf_node_eval_param": {
    "version": "1.0.0",
    "comp_id": "privacy/l_diversity:1.0.0",
    "attr_paths": ["k", "l", "qi_cols_json", ...],
    "attrs": [ ... ],
    "inputs": [ ... ],
    "output_uris": [ ... ],
    "checkpoint_uri": "..."
  }
}
```

对 Kuscia 而言，`sf_node_eval_param` 就是一个**普通的 JSON 字符串/字节块**。Kuscia 不会尝试去理解 `comp_id` 为什么是 `privacy/l_diversity:1.0.0`，也不会去理解 `k`/`l` 代表什么业务含义。它只负责：

- 把这份配置作为文件挂载到任务 Pod 内（`/kuscia/task-config.conf`）。
- 根据 `sf_cluster_desc` 中的参与方信息创建网络连接。
- 等 Pod 退出后收集状态。

真正的组件语义解析发生在 SecretFlow 容器内部，由 `secretflow.kuscia.entry` 读取 `task-config.conf` 后调用 `comp_eval(...)` 完成。

### 2. Kuscia 的 CRD 是通用抽象

Kuscia 的核心 CRD 包括：

| CRD | 作用 | 是否感知组件类型 |
|-----|------|------------------|
| `KusciaJob` | 一次作业的顶层抽象 | ❌ 不感知 |
| `KusciaTask` | 作业下的任务，可按参与方拆分 | ❌ 不感知 |
| `DomainData` | 数据资产元数据（URI、类型、列信息等） | ❌ 不感知具体算法 |
| `DomainDataSource` | 数据源配置 | ❌ 不感知 |
| `DomainRoute` | 跨域路由 | ❌ 不感知 |

这些 CRD 的字段都是通用的：任务 ID、参与方、镜像名、资源限制、输入输出数据引用等。无论跑的是 `l_diversity`、`differential_privacy` 还是 `ss_lr`，CRD 结构完全相同。

### 3. 组件实现通过 SecretFlow AppImage 注入

Kuscia 调度任务时，只关心容器镜像名（AppImage）。对本次开发而言，就是：

```text
secretflow/sf-privacy-dev:1.15.0.dev-privacy
```

只要这个镜像里包含了 `privacy/l_diversity` 的 Python 实现和组件注册信息，Kuscia 就能调度它。新增组件时：

- **不需要**修改 Kuscia 二进制；
- **不需要**修改 Kuscia 的 CRD；
- **不需要**修改 Kuscia 的调度逻辑；
- 只需要确保 Kuscia Lite 节点本地镜像仓库中有新版 AppImage（通过 `docker save ... | kuscia image load` 导入）。

### 4. DataMesh 也是数据语义无关的

任务执行过程中，SecretFlow 会通过 DataMesh 读取/写入 `DomainData`。DataMesh 只关心：

- 数据在哪里（`datasource_id` / `uri`）；
- 数据格式（CSV/ORC/ODPS 等）；
- 谁有权限访问（`DomainDataGrant`）。

它并不关心这份数据是被 `l_diversity` 消费还是被 `differential_privacy` 消费。数据 schema 和类型由 SecretFlow 组件自身在 `evaluate()` 中解析。

### 5. 什么时候才需要改 Kuscia？

如果新增组件引发了以下变化，才需要考虑 Kuscia 侧改动：

- 需要新的 CRD 或扩展 CRD 字段（例如新的任务依赖模型、新的网络策略）。
- 需要新的调度语义（例如 GPU 资源、TEE 节点亲和性、跨域通信新协议）。
- 需要新的数据访问模式（例如 DataMesh 目前不支持的新存储后端）。
- 需要修改任务 Pod 的生命周期或注入方式。

对于普通的“新增一个算法组件”，以上都不涉及，因此 Kuscia 源码无需变更。

### 一句话总结

> **Kuscia 是“通用集装箱调度器”，SecretFlow 是“货物（算法组件）”，SecretPad 是“下单系统”。新增货物只需要把货物装进集装箱（镜像）并更新下单系统的商品目录（组件 JSON/前端模板），不需要改码头（Kuscia）。**

## 5、开发版与发布版的端口配置及代理资源说明

本节说明本地二次开发环境（dev）与生产/部署环境（prod）在端口和代理配置上的差异，避免在联调时把“开发端口”和“生产端口”混用。

### 5.1 本地开发版端口（非 Docker 启动）

使用 `scripts/dev-start.sh` 或 `scripts/run-all-no-docker.sh` 一键拉起时，各服务默认监听以下端口：

| 服务 | 端口 | 说明 |
|------|------|------|
| SecretPad 前端 dev server | **8000** | Umi dev，默认会把 `/api` 代理到后端 8080 |
| SecretPad 后端 HTTP | **8080** | `server.http-port`，前端 dev 代理目标 |
| SecretPad 后端 HTTPS | **8443** | `server.port`，生产模式下的安全入口 |
| SecretPad 内部 API | **9001** | `server.http-port-inner`，部分内部调用使用 |
| KusciaAPI gRPC | **18083** | SecretPad 后端通过它创建/查询 Kuscia Job |
| Kuscia Gateway 公共 | **18080** | 外部访问入口 |
| Kuscia Envoy 内部 | **13081** | 数据面通信，DataMesh 跨 Pod 通信经过这里 |

开发模式下，**前端 → 后端**的流量路径是：

```text
浏览器 http://localhost:8000
    └── Umi dev proxy (/api/*)
        └── http://localhost:8080
            └── SecretPad 后端
                └── gRPC 127.0.0.1:18083
                    └── Kuscia Master
```

因此本地开发时，浏览器只需要访问 `http://localhost:8000`，不需要直接访问 8080/8443。

### 5.2 前端 dev 代理配置

Umi 的 dev 代理通常通过项目根目录下的 `.env` 或 `config/proxy.ts` 配置：

```bash
# secretpad/frontend-src/.env（若存在）
PORT=8000
UMI_DEV_PROXY_TARGET=http://127.0.0.1:8080
```

如果后端不是本机 8080（例如在容器或远程机器上），修改上述 `UMI_DEV_PROXY_TARGET` 后重新启动前端即可。

### 5.3 发布版（生产/Docker 部署）端口

生产环境通常使用 Kuscia 官方部署脚本或 SecretPad 一体化部署，端口与开发环境不同。Kuscia 相关默认端口如下：

| 服务 | 默认端口 | 说明 |
|------|----------|------|
| KusciaAPI HTTP external | **8082** | 外部系统 REST 访问 |
| KusciaAPI gRPC | **8083** | 外部系统 gRPC 访问；非 Docker 开发模式映射为 18083 |
| KusciaAPI HTTP internal | **8092** | 集群内部使用 |
| DataMesh HTTP | **8070** | DataMesh REST |
| DataMesh gRPC | **8071** | DataMesh gRPC + Arrow Flight |
| ConfManager HTTP | **8060** | 证书/配置管理 HTTP |
| ConfManager gRPC | **8061** | 证书/配置管理 gRPC |
| Reporter HTTP | **8050** | 任务状态上报 |
| Transport gRPC | **9090** | 跨域传输 |
| Gateway public | **1080** | 公网入口 |
| Gateway internal | **80** | 集群内部入口；非 Docker 开发模式映射为 13081 |

SecretPad 后端连接 Kuscia 时，通过以下环境变量指定地址：

```bash
export KUSCIA_API_ADDRESS=127.0.0.1      # KusciaAPI gRPC 地址
export KUSCIA_API_PORT=18083             # 开发模式；生产若使用官方部署则常为 8083
export KUSCIA_GW_ADDRESS=127.0.0.1:13081 # 开发模式；生产常为 127.0.0.1:80 或 nginx 入口
export KUSCIA_PROTOCOL=notls             # 开发模式关闭 mTLS；生产通常为 tls
```

### 5.4 后端 application 中的端口配置

SecretPad 后端的 `application.yml`（位于 `secretpad/config/`）中：

```yaml
server:
  port: 8443                 # HTTPS 端口
  http-port: 8080            # HTTP 端口（开发模式前端代理目标）
  http-port-inner: 9001      # 内部 API 端口
```

生产部署若只需要 HTTPS，可将 `http-port` 关闭或改为与 `port` 一致，并通过 nginx/Envoy 做统一入口。

### 5.5 代理资源与跨域说明

- **本地 dev**：前端 Umi dev server 自带代理，不存在浏览器跨域问题。
- **生产环境**：前端静态资源通常由 nginx 或 Kuscia Gateway 托管，后端通过反向代理暴露 `/api`。需要确保：
  - Gateway/nginx 把 `/api/v1alpha1/*` 转发到 SecretPad 后端；
  - CORS 配置允许前端域名；
  - mTLS 证书（`server.jks`、CA 证书）正确挂载；
  - 防火墙开放 Kuscia 所需端口（8082/8083/8070/8071/80/1080 等）。

### 5.6 常见踩坑

| 现象 | 原因 | 解决 |
|------|------|------|
| 前端页面能打开，但接口报 404 | Umi dev proxy 未把 `/api` 转发到后端 | 检查 `.env` 或 `config/proxy.ts` |
| 后端日志显示连不上 Kuscia | `KUSCIA_API_ADDRESS` / `KUSCIA_API_PORT` 配置错误 | 开发模式用 `127.0.0.1:18083`；生产用实际地址 |
| Kuscia Pod 启动后无法跨域通信 | Gateway/Envoy 端口未开放 | 检查 80/13081/1080/18080 等端口监听 |
| 生产环境前端报 CORS 错误 | 后端未配置跨域或 nginx 转发丢失 Origin | 在 nginx 或后端统一配置 CORS |
