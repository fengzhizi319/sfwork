## 1. 环境准备

|------|---------|------|
| JDK | 17 | SecretPad 后端编译与运行 |
| Maven | 3.8.8+ | Java 构建工具 |

conda env list | grep sf310

## 2. 一键启动本地开发环境

### 2.1 使用一键脚本（推荐）

进入 sfwork 根目录，执行：

```bash
cd /home/charles/code/sfwork
bash scripts/dev-start.sh
```

首次运行会自动完成：

1. 检测 JDK 17 / Maven / Node.js / pnpm / Docker / conda `sf310`
2. 基于 `secretflow/` 源码构建自定义 SecretFlow 镜像 `secretflow/sf-privacy-dev:1.15.0.dev-privacy`
3. 部署 Kuscia master + alice + bob 容器，并注册上述自定义镜像作为 AppImage
4. 生成证书、编译 SecretPad 后端 fat jar
5. 启动后端和前端开发服务器

启动成功后会输出：

```text
🌐 前端开发服务器：http://localhost:8000
🔧 后端健康检查：http://localhost:8080/actuator/health
🔒 后端 HTTPS 地址：https://localhost:8443
🐳 自定义 SecretFlow 镜像：secretflow/sf-privacy-dev:1.15.0.dev-privacy
👤 登录账号：admin / 12345678
```

其他常用命令：

```bash
# 仅检查并自动安装/确认运行时
bash scripts/dev-start.sh --check

# 停止后端和前端（保留 Kuscia）
bash scripts/dev-stop.sh

# 同时停止 Kuscia 容器
bash scripts/dev-stop.sh --kuscia
```

### 2.2 环境变量说明

| 环境变量 | 默认值 | 说明 |
|----------|--------|------|
| `PRIVACY_IMAGE` | `secretflow/sf-privacy-dev:1.15.0.dev-privacy` | 自定义 SecretFlow 镜像 tag |
| `CONDA_ENV` | `sf310` | 构建 wheel 时使用的 conda 环境 |

示例：

```bash
PRIVACY_IMAGE="myregistry/sf-privacy:dev" CONDA_ENV="sf310" \
  bash scripts/dev-start.sh
```

## 3. 手动部署与镜像构建

如果你希望分步操作，或需要排查一键脚本中的某一步，可按以下流程手动执行。

### 3.1 构建二次开发 SecretFlow 镜像

基于 `secretflow/docker/privacy-dev/Dockerfile` 构建快速验证镜像：

```bash
cd /home/charles/code/sfwork/secretflow

source "$(conda info --base)/etc/profile.d/conda.sh"
conda activate sf310

# 1. 构建 wheel
rm -rf dist build
python -m build --wheel

# 2. 准备构建上下文
mkdir -p docker/privacy-dev
cp dist/secretflow-*.whl docker/privacy-dev/

# 3. 构建镜像
cd docker/privacy-dev

# 默认使用阿里云 PyPI 镜像源
docker build . -f Dockerfile -t secretflow/sf-privacy-dev:1.15.0.dev-privacy

# 若阿里云源出现下载中断/哈希校验失败，可切换到官方 PyPI：
# docker build . -f Dockerfile \
#   --build-arg PIP_INDEX_URL=https://pypi.org/simple/ \
#   -t secretflow/sf-privacy-dev:1.15.0.dev-privacy
```

产物镜像：

```text
secretflow/sf-privacy-dev:1.15.0.dev-privacy
```

验证镜像内组件已注册：

```bash
docker run --rm secretflow/sf-privacy-dev:1.15.0.dev-privacy \
  'python -c "from secretflow.component.core import Registry; \
              print(Registry.get_definition_by_id(\"privacy/l_diversity:1.0.0\").component_def.name)"'
```

> 注意：`ubuntu-base-ci` 镜像的 `ENTRYPOINT` 是 `/bin/bash -lc`，因此 `docker run` 时整个命令需要用**单引号**包成一组传入。

如需使用官方完整镜像构建方式（方式 B），请参考 `docs/privacy-component-deployment.md` 第 4.1 节。

### 3.2 部署 Kuscia（不部署容器版 SecretPad）

```bash
cd /home/charles/code/sfwork/secretpad

# 关键：通过环境变量指定自定义 SecretFlow 镜像
export SECRETFLOW_IMAGE="secretflow/sf-privacy-dev:1.15.0.dev-privacy"

# 部署 Kuscia master + alice + bob
bash scripts/install-kuscia-only.sh master -P notls
```

`-P notls` 表示不使用 mTLS，本地开发更简便。

部署成功后会创建以下容器（假设当前用户为 `charles`）：

| 容器名 | 说明 |
|--------|------|
| `charles-kuscia-master` | Kuscia Master 控制平面 |
| `charles-kuscia-lite-alice` | Lite 节点 alice |
| `charles-kuscia-lite-bob` | Lite 节点 bob |

默认端口映射（`master` 模式固定）：

| 服务 | 宿主机端口 | 容器端口 | 说明 |
|------|-----------|---------|------|
| Kuscia master gateway | 18080 | 1080 | 节点间认证鉴权 |
| Kuscia master KusciaAPI HTTP | 18082 | 8082 | HTTP API |
| Kuscia master KusciaAPI gRPC | 18083 | 8083 | SecretPad 后端连接此端口 |
| Kuscia master Envoy 内部端口 | 13081 | 80 | `KUSCIA_GW_ADDRESS` 指向此端口 |
| Kuscia lite alice gateway | 28080 | 1080 | alice 对外网关 |
| Kuscia lite alice KusciaAPI gRPC | 28083 | 8083 | alice API gRPC |
| Kuscia lite bob gateway | 38080 | 1080 | bob 对外网关 |
| Kuscia lite bob KusciaAPI gRPC | 38083 | 8083 | bob API gRPC |

验证自定义 AppImage 已注册：

```bash
docker exec -it ${USER}-kuscia-master kubectl get appimage
# 应看到 secretflow-image 使用的镜像为 secretflow/sf-privacy-dev:1.15.0.dev-privacy
```

查看 SecretFlow 镜像是否已导入 Kuscia：

```bash
docker exec -it ${USER}-kuscia-lite-alice kuscia image list
```

### 3.3 源码后端连接 Kuscia 容器

```bash
export KUSCIA_API_ADDRESS=127.0.0.1
export KUSCIA_API_PORT=18083
export KUSCIA_GW_ADDRESS=127.0.0.1:13081
export KUSCIA_PROTOCOL=notls
```

> 也可以修改 `/etc/hosts` 将 `root-kuscia-master`、`root-kuscia-lite-alice`、`root-kuscia-lite-bob` 指向 `127.0.0.1`，但使用环境变量更简单。

## 4. 后端服务启动

```bash

```

     -Dsun.net.http.allowRestrictedHeaders=true \

## 7. Nginx / Tomcat / Kuscia 端口配置逻辑

### 7.1 Tomcat 端口

| 配置项 | 默认端口 | 作用 |
|--------|---------|------|
| `server.port` | 443 / 8443 | HTTPS 主端口 |
| `server.http-port` | 8080 | HTTP API 端口，前端 dev 代理指向这里 |
| `server.http-port-inner` | 9001 | 内部 RPC 端口 |

### 7.2 Kuscia 端口映射

| 服务 | 容器端口 | Master 宿主机端口 | Alice 宿主机端口 | Bob 宿主机端口 | 说明 |
|------|---------|------------------|-----------------|---------------|------|
| Gateway | 1080 | 18080 | 28080 | 38080 | 节点间认证、任务路由 |
| KusciaAPI HTTP | 8082 | 18082 | 28082 | 38082 | HTTP 管理 API |
| KusciaAPI gRPC | 8083 | 18083 | 28083 | 38083 | SecretPad 后端连接此端口 |
| Envoy Internal | 80 | 13081 | 23081 | 33081 | 数据传输、任务调度内部通道 |

### 7.3 三者如何对接

1. **前端 → 后端**：`PROXY_URL=http://127.0.0.1:8080`
2. **后端 → Kuscia API**：`KUSCIA_API_ADDRESS=127.0.0.1`，`KUSCIA_API_PORT=18083`
3. **后端 → Kuscia Gateway**：`KUSCIA_GW_ADDRESS=127.0.0.1:13081`

## 8. 常见问题排查

### 8.1 `install-kuscia-only.sh` 仍拉取官方 SecretFlow 镜像

请确认调用前已正确导出环境变量：

```bash
export SECRETFLOW_IMAGE="secretflow/sf-privacy-dev:1.15.0.dev-privacy"
bash scripts/install-kuscia-only.sh master -P notls
```

`install-kuscia-only.sh` 已修改为支持通过环境变量覆盖镜像，且对于本地已存在的镜像会跳过拉取。

### 8.2 任务执行时提示找不到镜像

1. 确认自定义镜像已导入 Docker：
   ```bash
   docker images | grep sf-privacy-dev
   ```
2. 确认镜像已导入 Kuscia：
   ```bash
   docker exec -it ${USER}-kuscia-lite-alice kuscia image list
   ```
3. 确认 AppImage 已注册：
   ```bash
   docker exec -it ${USER}-kuscia-master kubectl get appimage -o yaml
   ```

### 8.3 后端连接 Kuscia 失败

```bash
export KUSCIA_API_ADDRESS=127.0.0.1
export KUSCIA_API_PORT=18083
export KUSCIA_GW_ADDRESS=127.0.0.1:13081
export KUSCIA_PROTOCOL=notls
```

检查端口：

```bash
ss -tlnp | grep -E ':(18083|13081)\b'
docker ps | grep kuscia
```

### 8.4 后端端口冲突

参考 [无 Docker 运行说明](../无docker运行说明.md) 中的端口冲突排查。

### 8.5 构建 wheel 时依赖缺失

```bash
conda activate sf310
pip install --upgrade build setuptools wheel
```

## 10. 配置文件与资源详解

本节说明 SecretPad 运行所需的关键配置文件、资源文件的位置、作用、关键参数含义以及修改方式。二次开发过程中，若需要调整端口、数据源、组件显示、Kuscia 连接方式等，通常只需修改这些文件（或对应环境变量），无需改动代码。

### 10.1 `config/application.yaml`（基础配置）

位置：`secretpad/config/application.yaml`  
作用：Spring Boot 主配置文件，所有运行环境都会加载；特定 profile（如 `dev`、`edge`、`p2p`、`test`）会在此基础上覆盖同名配置。

| 配置分组 | 关键参数 | 默认值 | 含义 |
|---|---|---|---|
| `server` | `http-port` | `8080` | HTTP API 端口，前端 dev 代理默认指向此处 |
| `server` | `http-port-inner` | `9001` | 内部节点通信端口（部分同步接口免认证） |
| `server` | `port` | `443` | HTTPS 主端口；开发时常用 `-Dserver.port=8443` 覆盖 |
| `server.ssl` | `key-store` | `file:./config/server.jks` | HTTPS 证书密钥库 |
| `server.ssl` | `key-store-password` | `${KEY_PASSWORD:secretpad}` | JKS 密码，可通过环境变量 `KEY_PASSWORD` 覆盖 |
| `spring.datasource.default` | `jdbc-url` | `jdbc:sqlite:./db/secretpad.sqlite` | 业务数据库，默认 SQLite |
| `spring.datasource.quartz` | `jdbc-url` | `jdbc:h2:./db/secretpadQuartz.mv.db` | Quartz 定时任务专用 H2 数据源 |
| `spring.flyway.default.locations` | — | `filesystem:./config/schema/center` | Flyway 迁移脚本目录，随部署模式切换 |
| `kusciaapi` | `protocol` | `${KUSCIA_PROTOCOL:tls}` | KusciaAPI 全局协议：`tls` 或 `notls`/`grpc` |
| `kuscia.nodes[0]` | `domainId` / `host` / `port` | `kuscia-system` / `root-kuscia-master` / `8083` | Master 节点连接信息，可被 `KUSCIA_API_ADDRESS`、`KUSCIA_API_PORT` 覆盖 |
| `kuscia.nodes[1/2]` | — | alice / bob | Lite 节点连接信息；开发时通常只与 Master 直接交互 |
| `secretpad.platform-type` | — | `CENTER` | 平台架构：`CENTER`、`EDGE`、`P2P`/`AUTONOMY` |
| `secretpad.deploy-mode` | — | `${DEPLOY_MODE:ALL-IN-ONE}` | 部署模式：`MPC`、`TEE`、`ALL-IN-ONE` |
| `secretpad.node-id` | — | `kuscia-system` | 当前 SecretPad 实例所属节点 ID |
| `secretpad.gateway` | — | `${KUSCIA_GW_ADDRESS:127.0.0.1:80}` | Kuscia Gateway 地址，任务调度和数据传输使用 |
| `secretpad.auth` | `enabled` | `true` | 是否启用登录认证；`test` profile 关闭 |
| `secretpad.data.dir-path` | — | `/app/data/` | 用户上传文件与临时数据目录 |
| `secretpad.datasync` | `center` / `p2p` | `true` / `false` | 多节点数据同步模式开关 |
| `secretpad.version.*` | — | 多个镜像版本 | 用于注册 AppImage 时声明依赖镜像版本 |
| `secretpad.component.hide` | — | 若干组件 ID | 前端不显示的底层组件列表 |
| `secretpad.data-proxy.enabled` | — | `${DATAPROXY_ENABLE:true}` | 是否启用 Data Proxy |
| `secretpad.scql.enabled` | — | `${SCQL_ENABLE:true}` | 是否启用 SCQL 功能 |
| `sfclusterDesc.deviceConfig` | `spu` / `heu` | JSON 字符串 | SPU/HEU 安全计算协议与参数 |
| `sfclusterDesc.rayFedConfig` | `crossSiloCommBackend` | `brpc_link` | 跨节点通信后端 |
| `inner-port.path` | — | 4 条路径 | 可通过 9001 端口免认证访问的内部接口 |
| `ip.block` | `enable` / `list` | `true` / 多个 CIDR | SSRF 防护 IP 黑名单 |

修改方式：
- 推荐：通过环境变量覆盖，如 `KUSCIA_API_ADDRESS=127.0.0.1`。
- 也可直接编辑 YAML，但生产环境建议保留默认文件，仅通过环境变量注入差异配置。

### 10.2 多环境 Profile 配置

位置：`secretpad/config/application-dev.yaml`、`application-edge.yaml`、`application-p2p.yaml`、`application-test.yaml`

| 文件 | 生效方式 | 主要覆盖内容 |
|---|---|---|
| `application-dev.yaml` | `-Dspring.profiles.active=dev` | `platform-type: CENTER`；Kuscia 节点端口改为 `18083/28083/38083`；日志路径改为 `../log`；Gateway 默认 `127.0.0.1:18301`（一键脚本中通过 `KUSCIA_GW_ADDRESS=127.0.0.1:13081` 覆盖） |
| `application-edge.yaml` | `spring.profiles.active=edge` | `platform-type: EDGE`；Flyway 使用 `schema/edge`；只配置一个 Lite 节点；定义边缘节点转发/允许访问的 API 路径 |
| `application-p2p.yaml` | `spring.profiles.active=p2p` | `platform-type: AUTONOMY`；Flyway 使用 `schema/p2p`；开启 P2P 数据同步；配置 `unregister-mapping` 禁用中心化接口 |
| `application-test.yaml` | 单元测试 | 关闭认证、关闭 job 同步、使用 `../tmp/` 临时目录 |

切换 profile 示例：

```bash
java -Dspring.profiles.active=dev -jar target/secretpad.jar
```

### 10.3 组件定义与国际化资源

组件元数据：
- 位置：`secretpad/config/components/*.json`
- 加载配置项：`component.spec.location`，默认 `./config/components`
- 文件：
  - `secretflow.json` — SecretFlow 官方组件（含 `privacy/l_diversity:1.0.0` 等）
  - `secretpad.json` — SecretPad 平台组件（如 `read_data/datatable`）
  - `secretpad_tee.json`、`trustedflow.json`、`scql.json` — TEE、TrustedFlow、SCQL 组件

组件国际化：
- 位置：`secretpad/config/i18n/*.json`
- 加载配置项：`component.i18n.location`，默认 `./config/i18n`
- 文件名与 `components/` 一一对应，提供前端展示的中文/英文名称、参数说明等。

修改方式：新增或调整组件时，需同时修改 `components/` 与 `i18n/` 下对应 JSON，然后重启后端。仅修改 JSON 无需重新编译 Java。

### 10.4 Kuscia 节点序列化配置

位置：`secretpad/config/kuscia/{domainId}`（目录下为以域 ID 命名的序列化文件）

这些文件是后端根据 `kuscia.nodes` YAML 配置生成的 Java 序列化对象（`KusciaGrpcConfig`），用于缓存 Kuscia 连接信息。例如 `config/kuscia/alice/alice` 中包含 host、port、协议、证书路径等。

> **注意**：不建议手工修改这些序列化文件。若 Kuscia 连接信息变更，直接修改 `application*.yaml` 或对应环境变量，后端会在启动或重新加载时更新它们。

### 10.5 证书与 HTTPS 密钥库

位置：`secretpad/config/certs/` 与 `secretpad/config/server.jks`

| 文件/目录 | 作用 |
|---|---|
| `certs/ca.crt`、`ca.key`、`ca.srl` | 本地开发 CA |
| `certs/client.crt`、`client.pem`、`token` | Master 节点 KusciaAPI 客户端证书与 Token |
| `certs/alice/`、`certs/bob/`、`certs/tee/` | 各 Lite 节点的证书与 Token |
| `config/server.jks` | SecretPad HTTPS 服务证书密钥库 |

生成方式：执行 `secretpad/scripts/test/setup.sh`，会根据 `application*.yaml` 中配置重新生成上述证书与 JKS。

### 10.6 数据库迁移脚本

位置：`secretpad/config/schema/`

```text
schema/
├── center/       # CENTER 模式下的业务表迁移脚本
├── edge/         # EDGE 模式下的业务表迁移脚本
├── p2p/          # P2P/AUTONOMY 模式下的业务表迁移脚本
└── quartz/       # Quartz 定时任务表迁移脚本
```

每个目录下按 `V1__init.sql`、`V2__0.8.0.sql` 等 Flyway 版本号命名。Flyway 启动时会根据 `spring.flyway.default.locations` 自动执行未应用的脚本。**不要重命名或删除已有脚本**，否则会导致 Flyway 校验失败。

### 10.7 `ownerId`

位置：`secretpad/config/ownerId`

内容示例：

```text
kuscia-system
```

表示当前 SecretPad 实例对应的 Kuscia 域 ID，后端启动时会读取。在 EDGE/P2P 模式下应与 `NODE_ID` 环境变量保持一致。

### 10.8 前端代理配置

位置：`secretpad/frontend-src/apps/platform/.env`（不存在时手动创建）

开发模式下，Umi dev server 通过 `config/proxy.ts` 读取 `PROXY_URL`，将 `/api` 请求转发到后端：

```bash
echo "PROXY_URL=http://127.0.0.1:8080" > secretpad/frontend-src/apps/platform/.env
```

若后端 HTTP 端口改为其他值（如 `8081`），只需修改 `.env` 中的 `PROXY_URL` 并重启前端 dev server：

```bash
pnpm --filter secretpad dev
```
