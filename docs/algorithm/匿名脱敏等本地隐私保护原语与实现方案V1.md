| 技术               | 医疗数据分级映射       |
| ------------------ | ---------------------- |
| **差分隐私 (DP)**  | L3→L2（统计发布）      |
| **K-匿名**         | L3→L2（泛化后降级）    |
| **脱敏**           | L3→L2/L1（脱敏后降级） |
| **查询混淆 (QOL)** | L3/L4（查询意图保护）  |

总体架构定位：数据降级与意图保护的核心枢纽

在 C-Life 隐私计算平台的三阶段架构中，第一阶段（Privacy Gateway 预处理引擎）处于“安检门”与“过滤器”的核心地位。

本方案集成的四大关键隐私技术，其根本定位并非为了“把数据锁死”，而是为了实现**医疗数据要素的安全降维与意图隐匿**，从而打通数据出域流通的合规路径。

具体而言，四大技术实现的安全效能与架构定位如下：

#### 1. 脱敏与掩码 (Data Sanitization & Masking) —— **合规出域的基础**

- **安全效果：** **L3** \rightarrow **L2 / L1**（脱敏后安全降级）。
- **架构定位：** 作为数据流出的第一道防线。通过对敏感个人标识（PII，如身份证、姓名等 L3 数据）执行硬删除、正则掩码或假名化处理，将数据强制降级至 L2（机构运营级）或 L1（公开共享级），使其满足基本的合规出域或大屏展示要求。

#### 2. K-匿名 (k-Anonymity) —— **防御单点重识别**

- **安全效果：** **L3** \rightarrow **L2**（泛化后安全降级）。
- **架构定位：** 针对“准标识符”（如年龄、邮编、疾病），通过多维区间泛化（如将“28岁”泛化为“20-30岁”），确保数据集中任何一条记录至少隐藏在 k 个人的人群中。它有效抵御了外部数据链接攻击，使原本高风险的 L3 数据可以安全降级，满足联合科研与数据交付的标准。

#### 3. 差分隐私 (Differential Privacy, DP) —— **保护宏观统计的终极防线**

- **安全效果：** **L3** \rightarrow **L2**（统计级安全发布）。
- **架构定位：** 在进行跨机构医保统计、患病率计算或模型梯度聚合时，通过在结果中注入符合拉普拉斯/高斯分布的受控噪声，从数学原理上切断了“通过统计结果反推个人数据”的可能性。它将敏感的底层个体数据（L3）转化为高安全的宏观统计资产（L2）。

#### 4. 查询混淆 (Query Obfuscation Layer, QOL) —— **大模型时代的意图保护伞**

- **安全效果：** **L3 / L4**（高敏感查询意图保护）。
- **架构定位：** 并非对底层数据进行降级，而是针对“数据流向云端大模型”的场景。在业务系统或医生向外部 AI 发起包含高敏病种（L4）或个人特征（L3）的查询时，通过动态注入语义相似的“Dummy Queries（假查询）”，掩护真实查询意图，彻底防止外部不可信模型窃取我方的业务重心与高敏关注点。

**一句话总结架构价值：**

通过“脱敏、K-匿名、差分隐私”实现底层数据的**物理降级与流通解绑**；通过“查询混淆”实现云端协同的**意图隐形**，共同构筑了 C-Life 平台坚不可摧的安全底座。

# **差分隐私（Differential Privacy）**

## **算法介绍**

**差分隐私（Differential Privacy, 简称 DP）是目前数据隐私保护领域最严谨、最前沿的数学框架。与** k**-匿名通过“泛化”和“隐匿”来隐藏数据不同，差分隐私的核心思想是“引入受控的随机噪声”**。

它的核心承诺是：**无论某个具体个人的数据是否包含在数据集中，算法的输出结果都几乎保持不变。** 这样一来，攻击者就无法通过观察输出结果，反推或确认任何一个个体的具体信息。

以下是差分隐私的核心概念、数学原理以及常用机制的详细介绍。

###  差分隐私的核心概念

1. **相邻数据集 (Adjacent Datasets):** 假设有两个数据集 D_1 和 D_2，它们之间有且仅有一条记录不同（即相差一个人的数据），这两个数据集被称为相邻数据集。
2. **隐私预算 (**\epsilon**, Epsilon):** 差分隐私通过参数 \epsilon 来量化隐私保护的强度。

- \epsilon越小：注入的噪声越多，隐私保护越强，但数据的可用性（准确度）越低。
- \epsilon越大：注入的噪声越少，数据越准确，但隐私泄露的风险越高。

1. **敏感度 (**\Delta f**, Sensitivity):** 指的是在所有可能的相邻数据集中，增加或删除一条记录对查询函数 f 输出结果造成的最大改变范围。敏感度决定了我们需要加入多少噪声。

###  数学定义

一个随机算法 \mathcal{A} 被称为满足 \epsilon-差分隐私（纯差分隐私），如果对于所有相邻的数据集 D_1 和 D_2，以及算法 \mathcal{A} 输出空间的所有子集 S，都满足以下不等式：

Pr[\mathcal{A}(D_1) \in S] \le e^{\epsilon} \cdot Pr[\mathcal{A}(D_2) \in S]

这个公式的直观解释是：无论一个人是否参与了数据统计，统计算法得到某个特定结果的概率之比，被严格限制在 e^{\epsilon} 范围内。因为 e^{\epsilon} 接近于 1（当 \epsilon 很小时），这意味着攻击者无法通过观察输出来区分底层数据集到底是 D_1 还是 D_2。

###  实现差分隐私的三大经典机制

根据查询函数输出的数据类型不同，差分隐私通常采用以下三种机制来注入噪声：

#### 1. 拉普拉斯机制 (Laplace Mechanism)

- **适用场景:** 返回连续型数值的查询（如：计算平均工资、统计患病人数）。
- **原理:** 在真实的查询结果上，加上一个服从拉普拉斯分布的随机噪声。噪声的尺度与全局敏感度 \Delta f 成正比，与隐私预算 \epsilon 成反比。
- **公式:**

M(x) = f(x) + Lap(\frac{\Delta f}{\epsilon})

#### 2. 高斯机制 (Gaussian Mechanism)

- **适用场景:** 同样适用于数值型查询，常用于深度学习中的梯度加噪（如 DP-SGD）。
- **原理:** 添加服从高斯分布（正态分布）的噪声。它满足的是一种稍微放宽的差分隐私定义，称为 (\epsilon, \delta)-差分隐私，允许存在一个极小的概率 \delta 打破纯差分隐私的界限。

#### 3. 指数机制 (Exponential Mechanism)

- **适用场景:** 返回离散型/非数值型结果的查询（如：“数据集中最常见的血型是什么？”）。这种情况下不能直接加数字噪声（不能把“A型血”加上一个数字）。
- **原理:** 为每个可能的输出结果计算一个“打分”。分数越高（越接近真实答案）的选项，被算法选中的概率就呈指数级增大，但依然保留了输出其他较差选项的微小概率，从而实现隐私保护。

### 差分隐私的两大架构流派

1. **中心化差分隐私 (Centralized DP, CDP):**

- **架构:** 存在一个可信的数据收集中心。用户将真实的明文数据发给中心，中心在发布统计结果或训练模型时统一添加噪声。
- **特点:** 数据准确度高，加噪量相对较少。但存在单点信任风险（中心化服务器被黑客攻破则数据全泄露）。

1. **本地差分隐私 (Local DP, LDP):**

- **架构:** 不存在可信的第三方。用户在将数据发送给服务器**之前**，在自己的设备（本地）上先对数据进行加噪扰动。
- **特点:** 隐私安全性极高（服务器拿到的已经是假数据）。但为了抵消大量本地噪声，需要极其海量的用户数据才能统计出有用的宏观趋势（苹果和谷歌收集用户偏好数据时通常采用此方案）。

### 差分隐私的绝对优势

相比于上一代技术（如 k-匿名、l-多样性），差分隐私具有不可替代的优势：

- **免疫背景知识攻击:** k-匿名会被拥有丰富外部数据的攻击者破解，而差分隐私在数学上证明了：无论攻击者拥有多少外部背景知识，隐私承诺依然成立。
- **可组合性 (Composability):** 如果对同一个数据集进行多次差分隐私查询，整体的隐私损失是可以精确量化和累加的。这使得系统可以设置一个“总隐私预算池”，预算耗尽就拒绝查询，彻底防止隐私被“薅羊毛”。

## **算法实现方案设计**

采用 **中心化差分隐私（CDP）与本地差分隐私（LDP）相结合** 的混合架构。

**连续型数据（数值查询）**：

- **拉普拉斯机制**：`M(D) = f(D) + Lap(0, Δf/ε)`，适用于 COUNT、SUM、AVG 等单维统计
- **高斯机制**：`M(D) = f(D) + N(0, σ²)`，其中 `σ = Δf·√(2ln(1.25/δ))/ε`，适用于高维向量（联邦学习梯度）

**离散型数据（类别/编码）**：

- **指数机制**：从输出域 `R` 中以概率 `∝ exp(ε·u(D,r)/2Δu)` 选择最优输出，适用于最优 ICD-10 编码推荐
- **随机响应（Randomized Response）**：本地 LDP 场景，患者本地以概率 `p = e^ε/(e^ε+1)` 翻转布尔回答

**隐私预算管理**：

- 引入 **RDP（Rényi DP）组合定理** 替代朴素组合，支持跨查询的细粒度预算台账
- 建立 **Privacy Accountant**，每次查询后更新 `ε_累积`，触及阈值时触发熔断

以下是 **K-匿名算法**的完整实现方案、接口设计与核心代码。方案采用 **Mondrian 多维分区** 作为核心算法，支持**层次泛化**与**抑制**两种技术，并内建 **L-多样性** 和 **T-接近性** 检验，可直接嵌入医疗数据预处理流水线。

#### **核心设计**

| 模块                        | 职责                         | 关键技术                                 |
| --------------------------- | ---------------------------- | ---------------------------------------- |
| **GeneralizationHierarchy** | 管理每个 QI 维度的泛化层次树 | 数值区间化、分类型前缀截断、层次编码     |
| **MondrianPartitioner**     | 递归多维分区，生成等价类     | 中位数分割、信息损失增益优先             |
| **EquivalenceClass**        | 维护分区内的记录，执行泛化   | 区间合并、前缀提取、层次攀升             |
| **PrivacyValidator**        | 检验 K/L/T 约束              | K-匿名计数、L-多样性唯一值、T-接近性 EMD |
| **SuppressionHandler**      | 处理无法满足 K 的离群记录    | 小分区合并 / 记录删除                    |

#### **算法流程（Mondrian）**

```
输入：原始DataFrame，QI列表，SA列表，K/L/T参数
输出：匿名化DataFrame + 匿名化证明

1. 初始化：所有记录放入根分区
2. 递归分割：
   a. 选择最优QI维度（按信息损失增益或范围最大）
   b. 计算该维度中位数，将分区一分为二
   c. 若子分区均≥K，接受分割；否则回退
3. 终止条件：所有叶子分区大小≥K，或无法继续分割
4. 泛化：每个叶子分区内，将QI值泛化为统一表示（区间/前缀/层次）
5. 抑制：对仍<K的分区，执行合并到最近兄弟分区，或删除
6. 验证：输出前强制检验K/L/T，生成审计证明
```

- 当数据量 > 10 万行时，建议先对 QI 维度做离散化分桶，再运行 Mondrian，降低递归深度

# **K-匿名（K-Anonymity）**

## **算法介绍**

k-匿名（k-Anonymity）是一种数据脱敏技术，旨在防止数据记录被重新识别（Re-identification），从而保护个人隐私。它的核心思想是：**在发布的数据集中，对于任何一条记录，至少存在 k-1 条与其准标识符（Quasi-Identifiers, QIs）完全相同的其他记录。** 这样，即使攻击者掌握了某些背景知识，也最多只能将目标定位到 k 个人组成的一个组（即等价类），而无法确切识别出特定的个人。

### k-匿名的核心概念

1. **标识符 (Identifiers):** 能够直接且唯一确定个人身份的信息，如姓名、身份证号、手机号等。在发布数据前，这些信息通常会被直接删除或使用假名替换。
2. **准标识符 (Quasi-Identifiers, QIs):** 虽然不能直接确定个人身份，但当这些信息组合在一起，或者与外部数据源（如选民登记册）结合时，就有可能唯一识别出个人。常见的 QIs 包括：年龄、邮编、性别、职业等。k-匿名主要针对准标识符进行处理。
3. **敏感属性 (Sensitive Attributes):** 个人不希望被公开的信息，通常也是数据分析的主要目标，如疾病名称、收入水平、犯罪记录等。k-匿名通常不对敏感属性做处理，以保留其分析价值。
4. **等价类 (Equivalence Class):** 在数据集中，准标识符属性值完全相同的所有记录组成一个等价类。k-匿名要求每个等价类的大小至少为 k。

### 实现 k-匿名的主要技术

为了使数据集满足 k-匿名，通常采用以下两种技术：

1. **泛化 (Generalization):** 将具体的属性值替换为更宽泛的值。

- **区间化:** 例如，将具体年龄 `28` 泛化为年龄段 `20-30` 或 `[20, 30)`。
- **地址截断:** 例如，将精确地址 `深圳市南山区科技南十二路` 泛化为 `深圳市南山区`，或者将邮编 `518057` 泛化为 `5180`。

1. **抑制 (Suppression / 隐匿):** 如果某条记录的特征过于独特，无法与其他记录组成大小至少为 k 的等价类，且过度泛化会导致大量数据失真，则直接将该记录从发布的数据集中删除，或者将某些属性值置为 `*` (空值)。

### k-匿名的局限性

虽然 k-匿名有效地抵御了链接攻击（Linking Attack），但它存在一些固有的局限性：

1. **同质性攻击 (Homogeneity Attack):** 如果一个等价类内的所有记录具有相同的敏感属性值，那么即使达到了 k-匿名，隐私也会泄露。

- *例如：如果一个包含 5 人的等价类（即这 5 人的年龄段和邮编都一样）中，所有人的疾病都是“胃癌”，攻击者只要知道目标属于这个等价类，就能断定目标患有胃癌。*
- **解决方案:** 引入 l**-多样性 (**l**-Diversity)**，要求每个等价类中敏感属性至少有 l 个不同的值。

1. **背景知识攻击 (Background Knowledge Attack):** 攻击者可能利用额外的背景知识来排除等价类中的某些可能性，从而推断出敏感属性。

- *例如：攻击者知道目标属于一个包含“心脏病”和“前列腺癌”的等价类，且知道目标是女性，那么就可以排除“前列腺癌”，从而确定目标患有“心脏病”。*
- **解决方案:** 引入 t**-贴近性 (**t**-Closeness)**，要求每个等价类中敏感属性的分布与整个数据集的分布尽可能接近。

1. **维数灾难:** 当准标识符的数量增加时（高维数据），数据点在多维空间中变得稀疏，很难找到 k 个相似的记录。此时，为了达到 k-匿名，必须进行过度的泛化，导致数据效用（Utility）急剧下降。

## **实现方案**

以下是 **K-匿名算法** 的完整实现方案设计，涵盖算法选型、模块架构、接口定义、医疗场景适配及与 SecretFlow 的集成路径。

------

### **总体设计思路**

#### **算法选型**

针对医疗数据的高维稀疏特性（QI 维度通常包含年龄、性别、邮编、ICD-10 编码、就诊日期等），采用 **Mondrian 多维分区算法** 作为核心引擎，辅以 **层次泛化（Generalization Hierarchy）** 和 **抑制（Suppression）** 两种技术。

**选型理由**：

- **Mondrian** 通过递归选择最优 QI 维度进行中位数分割，相比 Datafly 的贪心泛化，在高维场景下信息损失更可控
- 支持增量式数据更新（新数据插入时仅需重新平衡局部子树）
- 天然适配医疗数据的区间型（年龄）和分类型（邮编/ICD-10）混合 QI 结构

#### **核心设计原则**

| 原则                           | 说明                                                         |
| ------------------------------ | ------------------------------------------------------------ |
| **字段级打标，表级就高不就低** | 每个 QI 列独立维护泛化层次，最终等价类取所有 QI 的泛化交集   |
| **隐私优先，效用兜底**         | 先满足 K/L/T 硬约束，再通过信息损失度量（LM）反馈优化分割策略 |
| **审计可追溯**                 | 每一步分割、泛化、抑制操作写入审计日志，支持合规溯源         |
| **与分级系统联动**             | K-匿名作为 L3→L2 的降级算子，由 Data Catalog 标签自动触发    |

##### `GeneralizationHierarchy`**（泛化层次管理器）**

**职责**：管理单个 QI 维度的泛化层次树，支持从精确值到最泛化值的多级攀升。

**接口方法**：

| 方法                                                         | 输入                                       | 输出           | 说明                             |
| ------------------------------------------------------------ | ------------------------------------------ | -------------- | -------------------------------- |

| `generalize(value, level)`                                   | 原始值、目标层次索引                       | 泛化后字符串   | 将值泛化到指定层次               |
| `get_common_ancestor(values)`                                | 值列表                                     | 共同泛化字符串 | 找到覆盖所有值的最小泛化级别     |
| `information_loss(values)`                                   | 值列表                                     | 浮点数 [0,1]   | 计算该组值的 LM 信息损失         |
| `get_level_for_values(values)`                               | 值列表                                     | 层次索引       | 自动推断需要攀升到第几层才能统一 |

**配置示例**（医疗场景）：

| `zipcode`        | 字符串 | `['exact', 'prefix-3', 'prefix-2', 'prefix-1', 'all']` | `518057` → `518***` → `51****` → `5*****` → `*`     |

### **与 SecretFlow 的集成方案**

#### **集成定位**

K-匿名作为 **数据预处理算子**，部署在 **PYU（明文计算设备）** 层，在数据进入联邦学习/MPC 之前完成本地泛化。

```
┌─────────────────────────────────────────────────────────────┐
│  SecretFlow 工作流（SecretPad/Kuscia 画布）                   │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────────┐   │
│  │  Data Source│ →  │ K-Anonymity │ →  │  FL / MPC /     │   │
│  │  (PYU本地)  │    │ Transformer │    │  DP Publisher   │   │
│  │             │    │ (PYU本地)   │    │ (PYU/SPU)       │   │
│  └─────────────┘    └─────────────┘    └─────────────────┘   │
│                              ↓                              │
│  ┌─────────────────────────────────────────────────────────┐│
│  │  Data Catalog 更新：                                      ││
│  │  原始标签 L3 → 经 K-Anon 泛化后 → 新标签 L2_K_ANON_K5   ││
│  │  血缘关系：原始表 → K-Anon 算子 → 匿名化表                ││
│  └─────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────┘
```

#### **PYU 组件封装**

**组件名称**：`KAnonymityTransformer`**设备类型**：`PYU`（各参与方本地执行，不跨节点通信）   **调度方式**：RayFed 本地 Actor 调用

**封装逻辑**：

1. **输入**：各参与方 PYU 接收本地 `DataFrame`（或 `VDataFrame` 的本地分片）
2. **配置下发**：通过 `Kuscia` 的 `ConfigMap` 或工作流参数下发 `K/L/T` 配置和泛化层次定义
3. **本地执行**：PYU 内实例化 `MondrianPartitioner`，对本地数据执行分区、泛化、抑制
4. **验证**：本地调用 `PrivacyValidator` 校验输出是否满足 K/L/T
5. **输出**：匿名化后的本地 `DataFrame`，标签回写本地 `Data Catalog`

**为什么放在 PYU 而非 SPU**：

- K-匿名是**数据预处理**操作，不需要多方密态计算
- 泛化操作需要读取原始明文值以计算区间/前缀，在 PYU 本地执行效率最高
- 输出为**脱敏后的明文数据**，可直接进入后续 FL/MPC 流程（此时数据已降级为 L2）

### **医疗场景适配方案**

#### **准标识符（QI）与敏感属性（SA）定义**

| 数据级别        | 典型 QI 列                                                 | 典型 SA 列                                       | 泛化策略                                                |
| --------------- | ---------------------------------------------------------- | ------------------------------------------------ | ------------------------------------------------------- |
| **L3 诊疗记录** | `age`, `gender`, `zipcode`, `admission_date`, `department` | `icd10`, `diagnosis_name`, `cost`, `medication`  | 年龄→10岁区间；邮编→前3位；日期→季度；ICD-10→大类       |
| **L3 检验指标** | `age`, `gender`, `test_date`                               | `glucose`, `creatinine`, `alt`, `wbc`            | 日期→月份；年龄→5岁区间                                 |
| **L4 敏感病种** | `age`, `gender`, `region`                                  | `hiv_status`, `mental_diagnosis`, `cancer_stage` | 年龄→20岁区间；地区→地级市；ICD-10→章节（如 B20-B24→B） |

#### **参数建议矩阵**

| 数据用途       | K 值 | L 值 | T 值 | 信息损失容忍 |
| -------------- | ---- | ---- | ---- | ------------ |
| 院内科研共享   | 5    | 2    | 0.2  | ≤ 0.3        |
| 跨院联邦学习   | 10   | 3    | 0.15 | ≤ 0.2        |
| 医保局统计发布 | 20   | 5    | 0.1  | ≤ 0.1        |
| 药企临床试验   | 5    | 2    | 0.2  | ≤ 0.25       |

#### **维数灾难应对**

当 QI 维度 > 5 时，Mondrian 算法可能因数据稀疏导致过度泛化。应对策略：

1. **QI 维度选择**：先执行互信息分析，剔除与 SA 关联弱、但重识别风险高的 QI（如 `department` 在某些医院可能唯一标识医生而非患者）
2. **层次预泛化**：在 Mondrian 前，先将高基数的 QI（如 `zipcode` 6 位）预泛化到 `prefix-3`，降低分割复杂度
3. **分桶预处理**：对连续型 QI（如 `cost`）先分桶为 10 个区间，再进入 Mondrian

1. **K-匿名证明**：`min_equivalence_class_size >= K`
2. **L-多样性证明**：`min_diversity_by_sa >= L`
3. **T-接近性证明**：`max_emd <= T`
4. **信息损失报告**：`LM_total = Σ(group_range / global_range) / N`
5. **抑制记录清单**：被删除/合并的记录 ID 列表（用于内部审计，不对外发布）

------

### **关键设计决策总结**

| 决策点         | 方案选择                      | 理由                                                         |
| -------------- | ----------------------------- | ------------------------------------------------------------ |
| **核心算法**   | Mondrian 多维分区             | 比 Datafly 更适合高维混合类型 QI，信息损失可控               |
| **泛化策略**   | 层次泛化（区间化 + 前缀截断） | 医疗数据天然适合区间（年龄）和层级（ICD-10）                 |
| **抑制策略**   | 小分区删除（非全局泛化）      | 避免为极少数离群点牺牲整体数据效用                           |
| **隐私增强**   | 内建 L-多样性 + T-接近性      | 防御同质性攻击和背景知识攻击                                 |
| **部署位置**   | SecretFlow PYU 本地预处理     | 泛化需要明文计算，PYU 效率最高；输出为脱敏明文，可直接进入 FL |
| **组件规范**   | SecretFlow V2 Component       | 支持前端画布拖拽，参数可视化配置                             |
| **与分级联动** | 输出自动打 `L2_K_ANON` 标签   | 下游 Gateway 根据标签自动放行到科研/统计场景                 |



# **脱敏（Data Sanitization）**

## **算法介绍**

**数据脱敏（Data Sanitization / Data Masking）是隐私计算和数据安全领域中最基础、应用最广泛的技术。与差分隐私（加噪）和** k**-匿名（泛化）改变数据分布不同，脱敏的核心目的是“在保持数据部分可用性或引用完整性的前提下，隐藏或替换掉直接的个人身份标识符（PII）”**。

脱敏可以分为**静态脱敏（SDM）**（将数据库克隆并清洗，用于开发/测试）和**动态脱敏（DDM）**（数据在数据库里是明文的，但在应用程序查询、运维人员查看时实时拦截并打码）。

以下是常见的脱敏算法与技术流派：

### 1. 掩码与截断 (Masking & Redaction)

这是最直观的脱敏方式，通常用于前端界面的展示（如您在网购、就医时看到的个人信息）。

- **部分掩码 (Partial Masking):** 使用特定字符（如 `*` 或 `X`）替换敏感字符串的一部分，保留原有格式。
- *手机号：* `13812345678` \rightarrow `138****5678`
- *身份证：* `110105199001011234` \rightarrow `110105********1234`（遮挡出生年月）
- **截断 (Truncation) / 置空:** 直接删除敏感数据，或者将其替换为默认值（如 `NULL` 或 `[REDACTED]`）。

### 2. 哈希与假名化 (Hashing & Pseudonymization)

当数据需要用于跨表/跨机构的联合分析（如计算 `JOIN`），但又不能泄露真实身份时，通常使用哈希。

- **单向哈希 (One-way Hashing):** 使用 SHA-256 等不可逆的密码学哈希函数处理数据。
- *原理:* `Hash("张三")` \rightarrow `b512d97e...`
- *加盐 (Salting):* 为了防止彩虹表攻击（黑客预先计算好所有手机号的哈希值进行反查），通常会加入一个随机的密钥（Salt），即 `Hash("张三" + Salt)`。
- **假名化 (Pseudonymization):** 使用一个随机生成的唯一标识符（Token/假名）替换真实 PII，并在一张受到极高安全级别保护的“映射表 (Token Vault)”中保存对应关系。只有拥有极高权限的系统才能将其还原。

### 3. 格式保留加密 (Format-Preserving Encryption, FPE)

在医疗和金融系统改造中，这是一种非常高级的脱敏技术。

- **痛点:** 很多老旧的业务系统（如 HIS 系统、银行核心系统）在数据库层面严格限制了字段的长度和数据类型。如果把 18 位数字的身份证号变成 64 位的字母哈希值，系统会直接报错崩溃。
- **原理:** FPE（如 AES-FF1 / FF3 算法）可以在加密数据的同时，**保持密文的长度和字符集与明文完全一致**。
- *例如:* 输入 16 位数字的银行卡号 `6222020200001234`，FPE 加密后变成另外一个看起来非常逼真的 16 位数字 `4192083749104821`。
- **特性:** 它是可逆的。掌握对称密钥的系统可以将密文解密回真实的卡号。

### 4. 替换与乱序 (Substitution & Shuffling)

常用于生成高质量的测试数据集。

- **字典替换 (Dictionary Substitution):** 建立一个“假名/假地址库”。遇到真实的“陈医生”，随机从库里抽一个“李医生”替换掉，保证数据看起来依然是真实的人类语言。
- **乱序 (Shuffling):** 将某一列的数据打乱重排。例如，把所有患者的“职业”这一列随机打乱。数据整体的宏观统计分布（如：程序员占 10%，教师占 15%）保持不变，但单条记录的对应关系被彻底破坏。

## **实现设计**

以下是 **脱敏（Data Sanitization）算法**的完整实现方案设计，涵盖双模架构（静态/动态）、五大技术流派的模块化封装、精细化接口定义，以及与 SecretFlow 隐私计算平台的深度集成路径。

------

### **总体设计思路**

#### **核心定位**

脱敏是隐私计算平台的**"安检门"**，与 K-匿名（泛化）、差分隐私（加噪）形成互补：

- **K-匿名**：改变 QI 的分布粒度，保护群体隐私
- **差分隐私**：改变统计结果的精确度，保护查询隐私
- **脱敏**：直接消除或替换**标识符本身**，保护个体身份

**设计原则**：

| 原则             | 说明                                                         |
| ---------------- | ------------------------------------------------------------ |
| **双模驱动**     | 静态脱敏（SDM）用于数据出域/测试/训练；动态脱敏（DDM）用于生产查询/运维审计 |
| **规则即策略**   | 字段级 JSON/YAML 配置化，支持按角色、按场景、按下游用途差异化脱敏 |
| **可逆性分级**   | 不可逆（掩码/哈希/截断）用于单向发布；可逆（FPE/Token Vault）用于系统间对接 |
| **格式保持优先** | 医疗 HIS 系统字段长度敏感，FPE 确保密文与明文格式完全一致    |
| **零拷贝执行**   | 正则预编译、向量化 apply、RE2 引擎，避免逐行 Python 循环     |

#### **医疗数据分级映射**

| 原始等级       | 脱敏目标      | 典型场景     | 技术组合                           |
| -------------- | ------------- | ------------ | ---------------------------------- |
| L3（PII）      | L2 或 L3 受限 | 院内科研共享 | 掩码 + FPE（可逆对接）             |
| L3（PII）      | L1 公开       | 公开统计报表 | 哈希不可逆 + 截断删除              |
| L4（敏感病种） | L3 脱敏       | 跨院科研     | 诊断编码泛化（K-Anon）+ 时间戳截断 |
| L5（基因）     | 禁止明文脱敏  | 任何场景     | 拒绝脱敏，强制入 TEE               |

### **核心算法模块设计**

#### **掩码与截断引擎（Masking & Redaction Engine）**

**职责**：前端展示级脱敏，不可逆，追求**人类可读但不可识别**。

**算法子策略**：

| 策略                        | 适用字段                   | 参数                                                         |
| --------------------------- | -------------------------- | ------------------------------------------------------------ |
| 部分掩码（Partial Masking） | 身份证号、手机号、银行卡号 | `keep_prefix`, `keep_suffix`, `mask_char`                    |
| 姓名掩码                    | 患者姓名、医生姓名         | `mask_mode`: `first_last`（张**）/ `middle`（*三）/* `all`*（***） |
| 地址截断                    | 住址、工作单位             | `truncate_level`: `province` / `city` / `district` / `street` |
| 置空（Redaction）           | 高度敏感且无需保留格式     | `replacement`: `NULL` / `[REDACTED]` / `*`                   |

**医疗场景映射**：

- 身份证号：`110105199001011234` → `110105********1234`（保留前6位地域码+后4位校验，中间12位掩码）
- 手机号：`13812345678` → `138****5678`
- 姓名：`张三丰` → `张**`（医生端可见）/ `***`（外部端）
- 地址：`深圳市南山区科技南十二路18号` → `深圳市南山区***`（科研端）/ `深圳市***`（外部端）

####  **哈希与假名化引擎（Hashing & Pseudonymization Engine）**

**职责**：跨表/跨机构联合分析时的身份隔离，支持**一致性关联**但不可逆向。

**算法子策略**：

| 策略                         | 适用场景                     | 关键参数                                                |
| ---------------------------- | ---------------------------- | ------------------------------------------------------- |
| 单向哈希（HMAC-SHA256）      | 不可逆场景（统计、公开）     | `salt`, `iterations`                                    |
| 加盐哈希（Salted Hash）      | 防彩虹表攻击                 | `salt_source`: `global` / `per_namespace` / `per_field` |
| 确定性假名化（Tokenization） | 可逆场景（跨系统患者ID关联） | `token_format`: `uuid` / `numeric` / `hash_prefix`      |

**Token Vault 设计**：

- **物理隔离**：Token Vault 部署于独立安全域（独立 VPC/TEE），与脱敏引擎分离
- **访问控制**：查询真实 ID 需**双人授权 + 审计日志**
- **生命周期**：Token 设置 TTL（如 90 天），过期自动失效
- **命名空间隔离**：不同医院/项目使用不同 namespace，防止跨域碰撞

**医疗场景映射**：

- 患者身份证号 → `HMAC(salt, id_card)` → `a3f7b2d8...`（16位截断，用于跨院科研主键）
- 病历号 → Token Vault 发行 `TKN-2024-8a7f3e2d` → 仅 HIS 内部系统可逆解析

#### **格式保留加密引擎（FPE Engine）**

**职责**：老旧 HIS/EMR 系统对接，**密文与明文格式完全一致**（长度、字符集、数据类型）。

**技术选型**：

- **算法**：AES-FF1（NIST SP 800-38G）或 FF3-1，优先 FF1（安全性经充分验证）
- **Tweak**：每个字段/每个患者/每个时间周期使用不同 Tweak，防止相同明文产生相同密文
- **字符集**：数字保留（`0-9`）、字母保留（`A-Z,a-z`）、混合保留（`A-Z,a-z,0-9`）

**医疗场景映射**：

- 医保卡号（19位数字）：`1234567890123456789` → FPE → `8472936105847263948`（19位数字，系统无感）
- 病历号（10位字母数字）：`MR20240001` → FPE → `MR9X7K2P8Q`（10位，格式保留）
- 身份证号（18位）：FPE 后仍为 18 位，但前 6 位地域码可能变为无意义数字，需评估业务影响

**密钥管理**：

- FPE 主密钥由 **HEU 设备分片存储**（SecretFlow HEU），解密需多方 MPC 授权
- 支持密钥轮换：旧密文批量重加密，Tweak 版本号递增

#### **替换与乱序引擎（Substitution & Shuffling Engine）**

**职责**：生成高质量测试数据集，保持**统计分布不变**但个体关联破坏。

**算法子策略**：

| 策略                                | 适用场景                     | 约束                                            |
| ----------------------------------- | ---------------------------- | ----------------------------------------------- |
| 字典替换（Dictionary Substitution） | 测试环境姓名、地址、医院名称 | 替换库需与原始数据同分布（性别比例、地域分布）  |
| 乱序（Shuffling）                   | 测试环境职业、科室、民族     | 单列独立打乱，保持频数分布                      |
| 条件乱序（Conditional Shuffling）   | 多列关联场景                 | 按性别分组后分别乱序职业，保持性别-职业联合分布 |

**医疗场景映射**：

- 测试环境患者姓名：`张三` → 从百万级假名库抽样 `李明辉`（同性别、同地域姓氏分布）
- 测试环境医生姓名：`陈主任` → `王主任`（从医师职称库替换）
- 科室列乱序：原始 `内科30%、外科25%、儿科15%...` → 乱序后比例不变，但患者-科室对应关系打破

| `intercept_sql(sql: str, user_role) -> rewritten_sql`        | 原始 SQL、用户角色                 | 改写后的 SQL | SCQL 集成，自动插入脱敏函数        |
| `register_callback(field_name, condition, callback)`         | 字段名、条件、回调函数             | 句柄         | 条件触发脱敏（如 L4 数据额外掩码） |

#### **FPE 专用接口（FPEEngine）**

| 方法                                                         | 输入                 | 输出       | 说明             |
| ------------------------------------------------------------ | -------------------- | ---------- | ---------------- |
| `encrypt(plaintext, key_id, tweak) -> ciphertext`            | 明文、密钥 ID、Tweak | 密文       | 格式保留加密     |
| `decrypt(ciphertext, key_id, tweak) -> plaintext`            | 密文、密钥 ID、Tweak | 明文       | 解密，需权限校验 |
| `rotate_key(old_key_id, new_key_id, batch_size) -> rotation_report` | 旧密钥 ID、新密钥 ID | 重加密报告 | 密钥轮换         |
| `validate_format(ciphertext, expected_format) -> bool`       | 密文、期望格式       | 是否匹配   | 格式校验         |

| ------------------------------------------------------- | ------------------------- | -------- | ------------ |
| `issue_token(real_id, namespace, ttl) -> token`         | 真实 ID、命名空间、有效期 | Token    | 发行假名     |

### **与 SecretFlow 的集成方案**

#### **集成定位**

脱敏引擎部署于 **PYU（明文计算设备）** 和 **Privacy Gateway（查询网关）** 两层，形成"**静态脱敏在入口，动态脱敏在出口**"的双闸门：

```
数据流入（医院本地）                    数据流出（查询响应）
     │                                      │
     ▼                                      ▼
┌─────────────┐                      ┌─────────────┐
│ 原始数据    │ ──PYU──▶ │ StaticDeidEngine │ ──▶ 脱敏 DataFrame
│ (L3)        │   本地    │  (SDM)           │     (L2/L3)
└─────────────┘                      └─────────────┘
                                              │
                                              ▼
                                       ┌─────────────┐
                                       │ K-Anon / DP │
                                       │ 后续处理     │
                                       └─────────────┘
                                              │
                                              ▼
                                       ┌─────────────┐
                                       │ SPU / SCQL  │
                                       │ 密态计算     │
                                       └─────────────┘
                                              │
                                              ▼
                                       ┌─────────────┐
                                       │ Privacy     │
                                       │ Gateway     │
                                       │ (DDM)       │
                                       │ 按角色动态  │
                                       │ 脱敏输出    │
                                       └─────────────┘
```

####  **PYU 层静态脱敏集成**

**组件名称**：`StaticDeidTransformer`**设备类型**：`PYU`（各参与方本地执行）   **调度方式**：RayFed 本地 Actor，数据不跨节点

**集成逻辑**：

1. **数据接入**：医院节点 PYU 从本地数据库（MySQL/Oracle/Hive）读取原始 DataFrame
2. **标签消费**：查询 Data Catalog，获取该表的字段级安全标签（如 `L3_PII_ID_CARD`、`L3_PII_NAME`）
3. **规则匹配**：根据标签自动匹配预置脱敏规则（ID_CARD → 掩码/哈希/FPE）
4. **本地执行**：PYU 内调用 `StaticDeidEngine.sanitize()`，逐列向量化脱敏
5. **输出分流**：
   - 不可逆脱敏（掩码/哈希）→ 直接进入下游 K-Anon/DP/FL 流程
   - 可逆脱敏（FPE/Token）→ FPE 密文参与计算，Token 映射关系本地落盘 Vault
6. **血缘注册**：在 Data Catalog 中建立 `原始表 → StaticDeidTransformer → 脱敏表` 血缘边

**为什么放在 PYU**：

- 脱敏需要读取原始明文 PII 以执行格式保留加密或掩码，必须在明文设备完成
- 静态脱敏是一次性预处理，不涉及时序要求，PYU 本地执行效率最高
- 输出为脱敏后明文，可直接进入后续联邦学习（此时数据已降级）

#### **Gateway 层动态脱敏集成**

**组件名称**：`DynamicMaskingMiddleware`**部署位置**：Privacy Gateway（API 网关中间件）   **触发时机**：查询结果返回前最后拦截

**集成逻辑**：

1. **查询拦截**：网关收到上层业务查询（SQL/API），解析涉及的数据资产和字段
2. **角色解析**：从 JWT/OAuth Token 中提取当前用户角色（`attending_doctor` / `researcher` / `external`）
3. **Catalog 查询**：查询 Data Catalog 获取字段的当前标签（注意：可能是 K-Anon/DP 处理后的标签）
4. **动态脱敏**：对结果集中每个字段值调用 `DynamicDeidEngine.mask_value()`，按角色选择算法
5. **SCQL 集成**：若底层使用 SecretFlow SCQL，网关自动改写 SQL，在查询计划中插入 `MASK()` 函数节点

**与 SCQL 的联动**：

```
-- 原始查询（研究员发起）
SELECT patient_name, id_card, diagnosis, cost FROM patient_records;

-- 网关改写后（SCQL 执行计划）
SELECT 
  MASK(patient_name, 'researcher') AS patient_name,  -- 输出: 张**
  HASH(id_card, 'research_salt') AS patient_id_hash,   -- 输出: a3f7b2...
  GENERALIZE(diagnosis, 'category') AS diagnosis,        -- K-Anon 泛化
  LAPLACE_NOISE(cost, 1.0) AS cost                       -- DP 加噪
FROM patient_records;
```

####  **HEU 层 FPE 密钥管理集成**

**核心设计**：FPE 主密钥不落地单点，由 **HEU（同态加密设备）** 分片管理：

```
┌─────────────────────────────────────────────────────────────┐
│  FPE Key Lifecycle                                           │
│                                                              │
│  生成：安全管理员在 TEE 内生成 FF1 主密钥                     │
│        ↓                                                     │
│  分片：密钥拆分为 3 份，分别存入 Hospital A/B/C 的 HEU 设备   │
│        ↓                                                     │
│  加密：使用 HEU 加密存储，任何单点无法恢复明文密钥              │
│        ↓                                                     │
│  使用：脱敏引擎调用 HEU.decrypt_shard() 获取部分密钥片段       │
│        多方 MPC 重组后执行 FPE 加解密（仅在内存中存在瞬时）      │
│        ↓                                                     │
│  轮换：旧密钥版本标记为 deprecated，新密钥分片重新生成          │
└─────────────────────────────────────────────────────────────┘
```

**集成点**：

- `FPEEngine` 初始化时，向 HEU 申请密钥分片，而非直接读取本地文件
- 解密操作（如 HIS 系统需要还原病历号）触发 HEU 多方授权流程
- 审计日志记录每次密钥使用（谁、何时、解密哪个字段、解密数量）

#### **Data Catalog 标签联动**

**脱敏后标签自动更新**：

| 原始标签               | 脱敏算法       | 新标签                           | 可逆性标记     |
| ---------------------- | -------------- | -------------------------------- | -------------- |
| `L3_PII_ID_CARD`       | 掩码（前6后4） | `L3_PII_ID_CARD_MASKED`          | `IRREVERSIBLE` |
| `L3_PII_ID_CARD`       | FPE            | `L3_PII_ID_CARD_FPE`             | `REVERSIBLE`   |
| `L3_PII_NAME`          | 哈希           | `L2_PII_NAME_HASHED`             | `IRREVERSIBLE` |
| `L3_PII_PHONE`         | 掩码           | `L3_PII_PHONE_MASKED`            | `IRREVERSIBLE` |
| `L4_SENSITIVE_DISEASE` | 时间戳截断     | `L4_SENSITIVE_DISEASE_TRUNCATED` | `IRREVERSIBLE` |

**血缘与冻结**：

- 脱敏操作作为算子节点写入血缘图谱：`原始表 → [Deid:mask] → 脱敏表`
- 若脱敏规则版本变更（如掩码从"前3后4"改为"前6后4"），触发下游表重新脱敏
- Token Vault 中的映射关系作为**独立血缘分支**，不与主数据血缘直接关联（防止逆向追踪）

#  **查询混淆层（Query Obfuscation Layer, QOL）**

## **算法介绍**

**核心定义**：在查询请求中混入虚假查询（Dummy Queries）和噪声结果，使得服务端无法推断用户的真实查询意图、查询对象和访问模式。

**查询混淆（Query Obfuscation）**，有时也被称为假查询注入（Dummy Query Injection），是一种为了保护用户“查询意图”而设计的隐私保护技术。

在传统的数据库查询或大模型问答中，即便连接是加密的（HTTPS），服务端依然能看到用户的查询内容（例如：医生查询了“张三的艾滋病治疗方案”）。如果服务端不可信，用户的查询偏好、关注点和隐私就会暴露。

查询混淆的核心思想是：**“隐藏在人群中（Hiding in the Crowd）”**。它通过在用户的真实查询中混入大量的、看起来非常逼真的“假查询（Dummy Queries）”，让服务端无法分辨哪个是真、哪个是假，从而保护真实的查询意图。

**医疗场景适配**：

- **患者跨院查询**：患者查询自己在某医院的就诊记录，混入其他医院的虚假查询，任何单一医院都无法确定患者是否查询了本院
- **药品溯源查询**：药企查询某批次原料，混入多个批次的假查询，供应商无法定位真实关注批次
- **科研统计查询**：研究员查询某病种发病率，混入多个无关病种的假查询，防止服务端推断研究焦点

**技术方案**：

**1. 查询膨胀（Query Expansion）**

- 将 1 个真实查询扩展为 `N = k + r` 个查询（`k` 为真实查询数，`r` 为假查询数）
- 假查询在**查询空间**中均匀分布，避免聚集暴露真实意图
- 假查询的生成必须服从与真实查询**不可区分的分布**，防止服务端通过异常检测过滤

**2. 假查询生成算法（Dummy Query Generator）**

- **基于数据分布采样**：从全局数据分布中采样假查询关键字，确保频率、长度、字符分布与真实查询一致
- **差分隐私假查询**：在查询频率中加入 Laplace 噪声，生成符合 `(ε, δ)`-DP 的假查询集合
- **语义保持假查询**：对医疗编码（ICD-10、药品编码）在相同大类中随机选择，保持语义合理性

**3. 结果混淆与本地过滤**

- 服务端对所有查询（真假混合）返回结果
- 客户端在**本地安全环境**中过滤假查询结果，仅保留真实查询对应的结果
- 结果数量也进行混淆：对假查询返回**空结果或随机噪声结果**，防止服务端通过结果大小推断真假

**4. 查询模式混淆（Access Pattern Obfuscation）**

- 引入 **ORAM（Oblivious RAM）** 或 **TEE 内查询混排**：在 TEE 中对查询序列进行随机置换，服务端只看到乱序的内存访问模式
- 结合 **差分隐私查询缓存**：在客户端维护本地缓存，以 `(ε, δ)`-DP 的方式决定是否真正发起查询，减少网络暴露

## **实现方案设计关键点**

### 查询混淆的算法设计难点

查询混淆看似简单（“多发几个假请求不就行了？”），但在实际工程落地中，设计高质量的混淆算法非常困难。核心挑战在于“如何让假查询看起来像真的”。

如果真实查询是关于罕见病（如“渐冻症”），而系统生成的假查询都是关于常见病（如“感冒”、“发烧”），那么拥有统计学知识的攻击者很容易通过词频分析和语义分析，把真实查询过滤出来。

一个优秀的查询混淆算法必须满足以下条件：

#### 1. 语义相似性 (Semantic Similarity)

假查询在语义上必须与真实查询处于同一个“上下文类别”中，但又不能太过接近以至于暴露真实意图。

- *反面教材：* 真实查询是“艾滋病抗病毒药物”，假查询是“最新款手机评测”。（太容易被识破）
- *正面教材：* 真实查询是“艾滋病抗病毒药物”，假查询是“乙肝抗病毒药物”、“丙肝治疗方案”。

#### 2. 查询频率分布一致性 (Frequency Distribution Consistency)

用户在实际系统中的查询频率是符合长尾分布（Zipf分布）的。假查询的生成频率也必须拟合这种分布。如果算法总是生成那些极少有人查询的冷门词汇作为假查询，攻击者一眼就能看出破绽。

#### 3. 用户画像一致性 (Profile Consistency)

如果一个用户的历史查询都是关于“儿科”的，突然某一次查询混入了一堆关于“老年痴呆”的假查询，这不仅不能保护意图，反而会暴露行为异常。

### C-Life 平台中的落地实现方案

可以采用**基于知识图谱与** k**-匿名思想结合**的查询混淆方案：

1. **构建本地医学语义库：** 在网关本地部署轻量级的 ICD-10 知识图谱或词嵌入向量（Word Embeddings）。
2. **拦截与聚类计算：** 当医生发起真实查询 `Q_real`（如“肺结节”）时，网关在本地图谱中寻找与“肺结节”距离相近、但在不同疾病分支上的 k-1 个节点（如“甲状腺结节”、“乳腺结节”）。
3. **生成 Dummy 集合：** 将真实查询和 k-1 个 Dummy 查询打包成一个 Batch。
4. **云端并发请求：** 网关向云端的 LLM 发起这 k 个查询。云端的大模型不知道哪个是真，只能全部作答。
5. **本地提取：** 网关收到 k 份回答后，丢弃其余 k-1 份，将“肺结节”的回答返回给医生前端。

### 优缺点分析

- **优点：**
  - **实现成本低：** 相比于极其复杂的密码学方案（如 ORAM 或同态加密 PIR），查询混淆完全基于应用层逻辑，不需要改造服务端的底层数据库，也不需要消耗巨大的计算力，响应速度极快。
  - **对服务端透明：** 外部大模型或云厂商完全感知不到查询混淆的存在，只需提供标准的 API 接口即可。
- **缺点：**
  - **带宽高、API 成本高：** 如果 k=5，意味着向大模型发起了 5 次请求，消耗了 5 倍的 Token 费用和网络带宽。这在商业化落地时是一笔不小的开销。
  - **抗统计攻击能力弱：** 无论 Dummy 算法设计得多精妙，如果服务端长时间监控一个用户的全量查询记录，结合先进的机器学习聚类算法，仍然有可能剥离出用户的真实长期偏好（即“背景知识攻击”）。

**总结：** 查询混淆是一种“用网络带宽/API成本换取隐私安全”的工程化妥协方案。在 C-Life 的三阶段架构中，它非常适合放在**第一阶段（Privacy Gateway）**，作为防范外部大模型收集用户提示词（Prompt）意图的一道轻量级防线。而在需要绝对安全的多方协作时（第二、三阶段），依然需要依赖 PSI 或 TEE 等硬核技术.

## **实现方案设计**

以下是 **查询混淆层（Query Obfuscation Layer, QOL）** 的完整实现方案与接口设计。该方案定位为 Privacy Gateway 的**应用层轻量级隐私防线**，通过"语义保持的假查询注入 + 本地过滤"实现查询意图保护，与 SecretFlow 的 PYU/SPU 设备层形成互补。

------

### **总体架构设计**

#### **核心定位**

QOL 部署在 **Privacy Gateway（API 网关层）**，作为客户端代理，位于用户真实请求与外部服务端（云端 LLM、数据目录、药品溯源库）之间。其核心职责是：

| 层级              | 组件                  | 防护对象                 | 技术强度         |
| ----------------- | --------------------- | ------------------------ | ---------------- |
| **QOL（应用层）** | 假查询注入 + 本地过滤 | 查询意图、访问模式       | 中（带宽换隐私） |
| **SPU（算法层）** | PSI / MPC PIR         | 查询内容、数据交集       | 高（密码学保障） |
| **TEE（硬件层）** | ORAM + 远程证明       | 内存访问模式、物理侧信道 | 极高（硬件隔离） |

**设计原则**：

- **语义不可区分**：假查询与真实查询在语义空间、频率分布、用户画像上保持统计一致
- **服务端零感知**：云端大模型/数据库无需任何改造，仅接收标准 API 请求
- **成本可控**：通过 DP 缓存、批量混排、语义相似度剪枝降低带宽开销
- **与分级联动**：L3 查询默认启用 QOL（k=3~5），L4 查询强制启用 QOL（k=5~10）+ SPU 密态混排，L5 查询拒绝 QOL（必须入 TEE）

### **核心模块设计**

#### 模块总览

| 模块                      | 职责                                          | 部署位置         |
| ------------------------- | --------------------------------------------- | ---------------- |
| `SemanticCorpus`          | 本地医学语义库（ICD-10 知识图谱、词嵌入向量） | Gateway 本地内存 |
| `DummyQueryGenerator`     | 基于语义距离和频率分布生成假查询              | Gateway 本地     |
| `QueryShuffler`           | 查询混排（随机置换顺序）                      | Gateway 本地     |
| `ResultFilter`            | 本地结果过滤，仅保留真实查询对应答案          | Gateway 本地     |
| `DPCacheManager`          | 差分隐私查询缓存，减少真实查询频率暴露        | Gateway 本地     |
| `AccessPatternObfuscator` | 查询时序混淆（时间抖动、批量聚合）            | Gateway 本地     |
| `CostController`          | 带宽/API 成本控制（k 值动态调整、语义剪枝）   | Gateway 本地     |

#### 本地医学语义库（SemanticCorpus）

**设计思路**：

- 在 Gateway 本地部署轻量级医学语义库，无需联网，避免云端推断语义意图
- 采用 **ICD-10 层次编码树** + **预训练医疗词嵌入（BioWordVec/PMC-Word2Vec，~100MB）** 双轨结构

**数据结构**：

| 组件           | 内容                                                | 用途                                               |
| -------------- | --------------------------------------------------- | -------------------------------------------------- |
| **ICD-10 树**  | 章节（A00-Z99）→ 大类（3位码）→ 亚类（4位码）→ 细目 | 语义保持：假查询与真实查询在相同章节/大类中采样    |
| **词嵌入向量** | 医学术语的 200-dim 稠密向量                         | 语义距离计算：找到与真实查询"相近但不同分支"的术语 |
| **查询频率表** | 全局查询日志的 Zipf 分布统计                        | 频率一致性：假查询的频率分布拟合真实长尾分布       |
| **用户画像库** | 按科室/角色聚合的历史查询主题分布                   | 画像一致性：儿科医生只混入儿科相关假查询           |

**更新机制**：

- 语义库通过离线包（月度）更新，不依赖实时云端同步
- 频率表基于本地审计日志滚动统计（滑动窗口 30 天）

#### 假查询生成器（DummyQueryGenerator）

**核心算法：基于语义距离 + k-匿名思想的查询混淆**

```
输入：真实查询 Qr，混淆参数 k，用户画像 P，语义库 S
输出：假查询集合 {Qd1, Qd2, ..., Qd(k-1)}

1. 语义解析：将 Qr 提取关键词 {w1, w2, ...}，映射到 ICD-10 节点或词嵌入向量
2. 候选生成：
   a. 在 ICD-10 树中，找到 Qr 所在节点
   b. 向父节点回溯 1-2 层，获取兄弟子树（同章节不同大类，或同大类不同亚类）
   c. 从兄弟子树中随机采样 (k-1) 个节点，作为语义锚点
   d. 对每个锚点，使用词嵌入向量在语义邻域（cosine similarity 0.6~0.85）内微调，生成自然语言假查询
3. 频率校准：
   a. 对每个候选假查询，查询本地频率表
   b. 若频率偏离全局 Zipf 分布 >20%，则重新采样（拒绝采样）
4. 画像校验：
   a. 检查假查询主题是否与用户画像 P 一致（如儿科医生不混入老年病）
   b. 若不一致，在画像主题内重新采样
5. 输出：{Qd1, ..., Qd(k-1)}，确保与 Qr 语义相近、分布一致、画像兼容
```

**医疗场景生成策略**：

| 真实查询            | 语义锚点策略                                       | 假查询示例           |
| ------------------- | -------------------------------------------------- | -------------------- |
| "肺结节随访方案"    | 同章节（R 呼吸系统）不同大类：甲状腺结节、乳腺结节 | "甲状腺结节随访方案" |
| "艾滋病抗病毒治疗"  | 同大类（B20-B24）相邻：乙肝抗病毒、丙肝抗病毒      | "乙肝抗病毒药物选择" |
| "BRCA1突变基因检测" | 同章节（C 肿瘤）不同基因：EGFR突变、KRAS突变       | "EGFR突变检测流程"   |
| "儿童糖尿病饮食"    | 同科室（儿科）不同病种：儿童肾病饮食、儿童甲亢饮食 | "儿童肾病低蛋白饮食" |

#### 查询混排器（QueryShuffler）

**职责**：打破查询时序关联，防止服务端通过请求顺序推断真实意图。

**策略**：

- **完全随机置换**：将真实查询与 (k-1) 个假查询随机排列，服务端无法通过位置推断
- **时间抖动**：每个查询的发送间隔加入 Laplace 噪声（`delay ~ Lap(0, 50ms)`），消除时序特征
- **批量聚合**：将多个真实查询（来自不同用户或同一用户的不同请求）合并为一个更大的混排批次，提升混淆强度

####  结果过滤器（ResultFilter）

**职责**：在本地安全环境中从云端返回的 k 个结果中，精准提取真实查询对应的结果。

**过滤机制**：

- **关键词匹配**：对返回结果进行本地关键词匹配，识别与真实查询最相关的结果
- **置信度打分**：使用本地轻量模型（如 10MB 级 BERT）对 k 个结果与 Qr 的相关性打分，取最高分
- **假查询结果丢弃**：对确认为假查询的结果，直接丢弃（不缓存、不持久化），防止本地侧信息泄露

**结果数量混淆**：

- 对假查询的结果，若服务端返回空结果或短结果，可能暴露其为假查询
- 解决方案：假查询的结果也进行本地"填充"或"截断"，使其长度分布与真实结果不可区分（或统一丢弃）

#### 差分隐私查询缓存（DPCacheManager）

**设计思路**：减少真实查询的网络暴露频率，即使攻击者监控网络流量，也无法通过查询频次推断用户关注焦点。

**机制**：

- **缓存命中概率**：以概率 `p = e^ε / (e^ε + 1)` 直接返回本地缓存结果，不发起真实网络查询
- **缓存失效策略**：缓存项设置 TTL，但 TTL 也加入 DP 噪声（`TTL ~ Lap(0, 1h)`），防止通过缓存失效模式推断
- **假查询缓存污染**：在缓存中混入假查询的"伪结果"，即使缓存被攻破，也无法区分真假

#### 查询模式混淆器（AccessPatternObfuscator）

**高级策略（L4 场景）**：

- **虚拟查询注入**：在用户无查询的时段，定期注入假查询（"心跳查询"），保持查询速率恒定，防止"突发查询暴露意图"
- **查询聚合**：将多个小查询合并为一个批量查询（Batch Query），减少请求次数和模式特征

#### 成本控制器（CostController）

**职责**：动态调整混淆强度，平衡隐私与成本。

**策略**：

| 场景           | k 值     | 成本优化                                                     |
| -------------- | -------- | ------------------------------------------------------------ |
| 日常查询（L3） | 3        | 语义剪枝：假查询与真实查询共享部分 API 上下文（如相同科室），减少 Token 差异 |
| 敏感查询（L4） | 5~7      | 批量混排：多个用户查询合并为一个大批次，分摊假查询成本       |
| 基因查询（L5） | 拒绝 QOL | 强制路由至 TEE，不消耗云端 API                               |

**语义剪枝优化**：

- 若假查询与真实查询在语义上高度相似（如"肺结节" vs "甲状腺结节"），可共享部分上下文 Token，降低 LLM 输出长度差异
- 对结构化查询（如数据库 SQL），假查询使用 `SELECT ... WHERE id IN (dummy_ids)` 的批量形式，减少 API 调用次数

### **与 SecretFlow 的集成方案**

#### 集成定位

QOL 部署在 **Privacy Gateway（应用网关层）**，与 SecretFlow 的 PYU/SPU/TEE 形成三层防御纵深：

```
用户请求
    │
    ▼
┌─────────────────────────────────────────┐
│  Privacy Gateway (QOL Agent)          │
│  ├─ 查询混淆（应用层）                   │
│  │   └─ 假查询注入 + 本地过滤            │
│  └─ 路由决策                             │
│      ├─ L3 查询 → 混淆后直接发往云端      │
│      ├─ L4 查询 → 混淆 + SPU 密态混排    │
│      └─ L5 查询 → 拒绝混淆，强制 TEE      │
└─────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────┐
│  SecretFlow 设备层                       │
│  ├─ PYU: 本地语义库存储、缓存管理         │
│  ├─ SPU: L4 查询的密态混排执行            │
│  └─ TEE: L5 查询的硬件隔离执行            │
└─────────────────────────────────────────┘
    │
    ▼
云端服务端（LLM / 数据库 / 数据目录）
```

#### 与 PYU 的集成

**集成点**：QOL 的本地语义库、用户画像、DP 缓存部署在各参与方的 PYU 设备上。

**逻辑**：

- 每个医院节点的 PYU 维护独立的 `SemanticCorpus`（ICD-10 树 + 词嵌入 + 频率表）
- 用户画像（医生科室、历史查询主题）存储在本地 PYU，不上传云端
- DP 缓存的命中决策在 PYU 本地执行，缓存内容不跨节点同步

**为什么放在 PYU**：

- 语义库需要明文访问以计算向量距离，PYU 是明文设备
- 用户画像属于敏感隐私数据，必须在本地存储
- 假查询生成涉及随机采样和概率计算，PYU 执行效率最高

#### 与 SPU 的集成（L4 增强）

**增强场景**：当查询涉及 L4 敏感数据（如 HIV、精神科）时，QOL 的"查询混排"在 SPU 内执行，防止网关本身被攻破后泄露真实查询索引。

**集成逻辑**：

1. Gateway 在 PYU 生成假查询明文
2. 将 k 个查询（真实+虚假）加密后传入 SPU
3. SPU 内部执行**密态混排**：在 MPC 协议内对查询顺序进行随机置换，任何一方（包括 Gateway 自身）都无法知道真实查询的位置
4. SPU 将混排后的密文查询发往云端，云端解密后执行
5. 云端返回的 k 个结果进入 SPU，SPU 根据预先约定的索引映射，将真实结果密态提取回 Gateway

**优势**：

- 即使 Gateway 被攻破，攻击者也无法从 SPU 的 MPC 执行日志中推断真实查询位置
- 与 QOL 的语义混淆形成"双层混淆"：语义上不可区分 + 位置上不可追踪

#### 与 TEE 的集成（L5 拒绝策略）

**L5 数据查询**（如基因位点查询）**拒绝使用 QOL**，原因：

- 基因查询的语义空间极度稀疏（BRCA1、TP53 等），假查询极易被识别
- 基因查询的结果通常很长（测序报告），假查询的成本极高（k 倍 Token 费用不可承受）
- 基因数据的隐私要求必须硬件级隔离

**路由策略**：

- Gateway 识别到查询标签含 `L5_GENOMIC` 时，直接拒绝 QOL 混淆
- 强制路由至 TEE 节点，在 TEE 内执行 ORAM 查询或本地预加载数据库查询

#### 与 Data Catalog 的联动

**标签驱动装配**：

- Data Catalog 中字段标签为 `L3_PII` 或 `L3_MEDICAL` 时，Gateway 自动装配 QOL（k=3）
- 标签为 `L4_SENSITIVE` 时，自动装配 QOL（k=5）+ SPU 密态混排
- 标签为 `L5_GENOMIC` 时，QOL 组件禁用，触发 TEE 路由

**血缘记录**：

- QOL 操作作为"虚拟算子"写入血缘图谱：`用户查询 → QOL 混淆 → 云端服务 → QOL 过滤 → 返回结果`
- 审计日志中记录批次 ID、假查询数量、是否命中缓存、成本消耗

### **成本、性能与可靠性**

#### 成本模型

| 指标           | 无混淆 | QOL (k=3) | QOL (k=5) | 优化后                  |
| -------------- | ------ | --------- | --------- | ----------------------- |
| API 调用次数   | 1x     | 3x        | 5x        | 批量混排可降至 1.5x~2x  |
| Token 消耗     | 1x     | ~3x       | ~5x       | 语义剪枝可降至 2x~2.5x  |
| 网络带宽       | 1x     | 3x        | 5x        | 结果压缩可降至 2x       |
| 延迟（端到端） | T      | T + 2Δt   | T + 4Δt   | 并行发送可控制在 T + Δt |

**成本优化策略**：

- **批量混排**：将多个用户的查询合并为一个大混排批次，分摊假查询成本
- **语义剪枝**：假查询与真实查询共享 API 上下文（如相同科室前缀），降低 LLM 输入长度
- **缓存优先**：DP 缓存命中率目标 40%~60%，减少真实查询重复消耗
- **分层 k 值**：L3 用 k=3，L4 用 k=5，避免一刀切

#### 性能指标

| 指标             | 目标值                      | 实现手段                                   |
| ---------------- | --------------------------- | ------------------------------------------ |
| 假查询生成延迟   | < 50ms                      | 本地语义库内存常驻，向量检索用 FAISS/Annoy |
| 混排 + 过滤延迟  | < 20ms                      | 纯本地内存操作                             |
| 云端并发请求     | 并行发送，无阻塞            | asyncio / aiohttp 并发                     |
| 整体端到端延迟   | < 2x 无混淆延迟             | 并行 + 缓存                                |
| 假查询不可区分度 | > 90%（语义 + 频率 + 画像） | 拒绝采样 + 分布校准                        |

#### 可靠性设计

| 风险           | 应对                                                     |
| -------------- | -------------------------------------------------------- |
| 云端服务限流   | 假查询降频：k 值自动下调至 2，或启用本地缓存             |
| 语义库过期     | 离线包更新失败时，降级为"同章节随机采样"，保证基本混淆   |
| 假查询生成失败 | 回退到"固定假查询模板库"（预置 1000 条高频医学查询）     |
| 本地过滤错误   | 若无法确定真实结果，返回"结果模糊，请重试"（安全侧保守） |

### **关键设计决策总结**

| 决策点                 | 方案选择                                        | 理由                                                         |
| ---------------------- | ----------------------------------------------- | ------------------------------------------------------------ |
| **核心算法**           | 语义保持假查询（ICD-10 树 + 词嵌入）            | 比纯随机假查询抗统计攻击能力强，比密码学 PIR 成本低          |
| **k 值策略**           | 动态分层（L3→k=3, L4→k=5, L5→拒绝）             | 与医疗数据分级系统联动，避免过度成本                         |
| **部署位置**           | Privacy Gateway（应用层）                       | 对服务端透明，无需改造云端；与 SecretFlow PYU/SPU/TEE 形成纵深 |
| **成本优化**           | 批量混排 + 语义剪枝 + DP 缓存                   | 将 k 倍成本压缩至 2~2.5 倍，商业可承受                       |
| **本地过滤**           | 关键词 + 语义匹配 + 置信度打分                  | 轻量、快速、可解释；错误时安全侧保守（拒绝返回）             |
| **与 SecretFlow 集成** | PYU 语义库 + SPU 密态混排（L4）+ TEE 拒绝（L5） | 充分利用 Device 抽象，明文/密态/硬件各司其职                 |
| **抗攻击能力**         | 承认局限性：长期监控下仍可能被剥离              | 明确边界，L4 长期查询建议升级至 SPU/TEE                      |

此方案将 QOL 定位为 **"用带宽换隐私的轻量级防线"**，在 C-Life 第一阶段（Privacy Gateway）作为外部大模型查询的标准配置，在第二、三阶段与 SPU/TEE 形成互补，覆盖从日常科研到基因计算的全场景查询隐私保护。

### 6.3 与 Data Catalog 的自动联动

```
def auto_assemble_privacy_pipeline(request, data_catalog):
    """
    根据 Data Catalog 标签自动装配隐私算子
    """
    tags = data_catalog.query_tags(request.data_assets)
    highest_level = max([tag.level for tag in tags])
    

    pipeline = [ ]

    
    if highest_level == 'L5':
        pipeline.append('TEE_MANDATORY')
        pipeline.append('ORAM_QUERY')  # L5 查询必须通过 ORAM
    elif highest_level == 'L4':
        pipeline.append('SPU_MPC')
        pipeline.append('QOL(semantic_dummy, ratio=5)')  # 查询混淆
        pipeline.append('K_ANON_CHECK')
    elif highest_level == 'L3':
        if request.purpose == 'statistical_publish':
            pipeline.append('DP_Preprocessor(ε=1.0)')
            pipeline.append('K_ANon_Transformer(K=5)')
        elif request.purpose == 'research_share':
            pipeline.append('Deid_Engine')
            pipeline.append('Dynamic_Mask(researcher)')
        elif request.purpose == 'cross_hospital_query':
            pipeline.append('QOL(distribution_dummy, ratio=3)')  # 查询混淆
            pipeline.append('Dynamic_Mask(resident)')
    
    return pipeline
```
